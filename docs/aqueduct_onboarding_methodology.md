# Aqueduct Onboarding Methodology

This document is the single source of truth for how Aqueduct 4.0 water-risk data was post-processed for the India Resilience Tool (IRT), how the raw Aqueduct layers were cleaned, and how Aqueduct values were transferred from HydroSHEDS Level 6 polygons onto the canonical Survey of India (SOI) `basin` and `sub-basin` polygons used in the dashboard.

The current onboarding tranche covers:

- Aqueduct baseline and future annual water-stress data
- India-only processing
- SOI hydro units only (`basin` and `sub-basin`)
- Dashboard integration under the hydro-only `Water Risk` bundle

## Why a special onboarding workflow was needed

Aqueduct could not be wired into the dashboard as-is for two separate reasons.

First, the raw Aqueduct `baseline_annual` layer was not geometrically reliable for direct use:

- its boundaries looked visibly inconsistent in GIS
- ArcGIS checks surfaced self-intersections and null geometries
- it was more segmented than the cleaner `future_annual` layer

Second, Aqueduct and IRT use different hydro boundary systems:

- Aqueduct is distributed on HydroSHEDS Level 6 polygons keyed by `pfaf_id`
- IRT renders SOI hydro polygons keyed as `basin` and `sub-basin`

Because of that, the onboarding workflow had to solve both:

1. clean and normalize Aqueduct on its own HydroSHEDS geometry
2. transfer Aqueduct values onto SOI hydro polygons in a scientifically consistent way

## Step 1: Inspect the raw Aqueduct layers

The first step was visual and structural inspection of the Aqueduct layers in ArcGIS Pro.

Observed behavior:

- `baseline_annual` displayed irregular internal boundaries
- `Check Geometry` reported defects such as self-intersections and null geometry
- `future_annual` looked much cleaner and behaved like a stable HydroSHEDS polygon layer

This led to the key methodological decision:

- do **not** use `baseline_annual` geometry as the canonical Aqueduct geometry
- use `future_annual` geometry as the clean HydroSHEDS reference
- use `baseline_annual` as an attribute source that must be normalized

This was a structural decision, not just a cosmetic one. Even if some baseline geometry could be repaired, the baseline layer was segmented differently and was not a clean one-row-per-HydroSHEDS-unit source.

## Step 2: Normalize baseline data to one row per `pfaf_id`

The central cleaning step was to rebuild baseline as a clean HydroSHEDS-level dataset using `pfaf_id` as the reference key.

### Why `pfaf_id`

`pfaf_id` is the stable HydroSHEDS identifier shared across Aqueduct products. It allowed us to:

- align baseline and future on the same hydrologic unit
- ignore broken baseline geometry
- collapse segmented baseline records into a single HydroSHEDS unit

Without this step, baseline and future would have remained structurally inconsistent.

### What the baseline cleanup tool does

[`tools/geodata/prepare_aqueduct_baseline.py`](../tools/geodata/prepare_aqueduct_baseline.py) performs the following:

1. Reads clean `future_annual` geometry from the Aqueduct geodatabase.
2. Reads `baseline_annual` from CSV, not from baseline geometry.
3. Canonicalizes `pfaf_id` values so IDs from the geodatabase and CSV join reliably.
4. Filters to India-only rows.
5. Drops invalid IDs such as `pfaf_id = -9999`.
6. Aggregates segmented baseline rows to one row per `pfaf_id`.
7. Joins those aggregated baseline values onto the clean `future_annual` geometry.

### Aggregation rule for baseline

Baseline rows were aggregated to `pfaf_id` using `area_km2` as the weight.

In practice this means:

- raw numeric baseline fields were area-weighted
- split-only identifiers such as `aq30_id`, `aqid`, and admin split columns were not carried forward
- category and label fields were not blindly transferred into the clean artifact

The result is:

- one clean India-only Aqueduct feature per `pfaf_id`
- clean HydroSHEDS geometry from `future_annual`
- normalized baseline attributes from the baseline CSV

### Important interpretation point

`baseline_clean_india.geojson` is **not** a repaired version of the original `baseline_annual` geometry.

It is:

- geometry from clean `future_annual`
- plus baseline attributes aggregated to `pfaf_id`

That distinction is essential.

## Step 3: Export the clean India-only Aqueduct geometry subsets

The cleanup stage produces two important India-only GeoJSONs.

### `baseline_clean_india.geojson`

Contains:

- one feature per `pfaf_id`
- clean HydroSHEDS geometry
- normalized baseline metric fields

Purpose:

- canonical baseline spatial source for further Aqueduct processing

### `future_annual_india.geojson`

Contains:

- India-only `future_annual` HydroSHEDS polygons
- preserved future annual Aqueduct attributes

Purpose:

- canonical future spatial source for later metric extraction and QA

## Step 4: Build the Aqueduct → SOI hydro crosswalk

Once Aqueduct was clean on HydroSHEDS Level 6, the next problem was the boundary mismatch with SOI hydro units.

Aqueduct is not displayed directly in the dashboard. Instead, it is transferred onto SOI hydro polygons in preprocessing.

This transfer is implemented in [`tools/geodata/build_aqueduct_hydro_crosswalk.py`](../tools/geodata/build_aqueduct_hydro_crosswalk.py).

### Conceptual approach

We did **not** try to force Aqueduct polygons to become SOI polygons.

Instead, we built a spatial overlap table that records:

- how much each Aqueduct `pfaf_id` polygon overlaps each SOI basin
- how much each Aqueduct `pfaf_id` polygon overlaps each SOI sub-basin

Those overlap areas then become weights for later metric transfer.

### Inputs

The crosswalk builder uses:

- clean India Aqueduct polygons from `baseline_clean_india.geojson`
- canonical SOI basin polygons
- canonical SOI sub-basin polygons

Only Aqueduct geometry is needed here. Metric transfer happens later.

### CRS handling

All source layers are stored in `EPSG:4326`, which is appropriate for storage and rendering but not for area calculation.

For overlap analysis, the tool reprojects both Aqueduct and SOI layers into:

- `EPSG:6933`

This is the analysis CRS used for:

- polygon intersection
- source polygon area calculation
- target polygon area calculation
- overlap area calculation

### Why `EPSG:6933`

The transfer is area-weighted, so it must be performed in an equal-area projection.

`EPSG:6933` was chosen because:

- it is equal-area
- it is already used by the repo’s polygon crosswalk methodology
- it gives scientifically defensible overlap weights without creating a second crosswalk methodology in the codebase

Stored and rendered geometries remain in `EPSG:4326`. Only the overlap math is done in `EPSG:6933`.

### Exact crosswalk method

For each run, the tool:

1. Loads Aqueduct polygons keyed by `pfaf_id`.
2. Loads SOI basins or sub-basins.
3. Reprojects both sides into `EPSG:6933`.
4. Computes native polygon areas:
   - `pfaf_area_km2`
   - `basin_area_km2` or `subbasin_area_km2`
5. Intersects the two polygon sets using `geopandas.overlay(..., how="intersection")`.
6. Computes `intersection_area_km2` for every overlap polygon.
7. Drops zero-area overlaps caused by edge-only or point-only touches.
8. Groups duplicate overlap fragments by identity keys and sums intersection area.
9. Calculates overlap fractions on both the Aqueduct and SOI sides.

### Meaning of the overlap fractions

For basin crosswalk rows:

- `pfaf_area_fraction_in_basin = intersection_area_km2 / pfaf_area_km2`
- `basin_area_fraction_in_pfaf = intersection_area_km2 / basin_area_km2`

For sub-basin crosswalk rows:

- `pfaf_area_fraction_in_subbasin = intersection_area_km2 / pfaf_area_km2`
- `subbasin_area_fraction_in_pfaf = intersection_area_km2 / subbasin_area_km2`

These fractions answer two different QA questions:

- what share of the Aqueduct polygon falls inside the SOI unit?
- what share of the SOI unit is covered by the Aqueduct polygon?

### What one crosswalk row means

One row in `aqueduct_subbasin_crosswalk.csv` means:

- Aqueduct polygon `pfaf_id = X`
- overlaps SOI sub-basin `subbasin_id = Y`
- by `intersection_area_km2 = Z`
- the overlap covers a known share of the Aqueduct polygon
- and a known share of the SOI sub-basin

That row later becomes one weighted contribution when SOI sub-basin Aqueduct values are computed.

## Step 5: Transfer Aqueduct metrics onto SOI basin and sub-basin units

The crosswalk itself stores geometry relationships only. It does not yet compute dashboard metric values.

The actual transfer happens in [`tools/geodata/build_aqueduct_hydro_masters.py`](../tools/geodata/build_aqueduct_hydro_masters.py).

### Inputs

The master-builder reads:

- `baseline_clean_india.geojson`
- `future_annual_india.geojson`
- `aqueduct_basin_crosswalk.csv`
- `aqueduct_subbasin_crosswalk.csv`

### Current metric mapping

For the current onboarding tranche:

- baseline water stress comes from `bws_raw`
- future water stress comes from `*_ws_x_r`

The dashboard-facing metric slug is:

- `aq_water_stress`

### Transfer rule

Each Aqueduct value is transferred to the SOI unit using overlap area as the weight:

\[
\text{SOI metric value} =
\frac{\sum(\text{Aqueduct value} \times \text{intersection area})}
{\sum(\text{intersection area})}
\]

In other words:

- every overlapping `pfaf_id` contributes to a basin or sub-basin
- larger overlap patches contribute more
- the final SOI value is an area-weighted mean of the contributing Aqueduct polygons

### Why this approach was chosen

This approach is more defensible than:

- centroid assignment
- nearest-neighbor transfer
- dominant-overlap-only assignment

because the boundary systems do not align one-to-one. Many SOI units overlap multiple HydroSHEDS polygons and vice versa. Area-weighted overlap preserves those partial relationships instead of collapsing them arbitrarily.

### Output master files

The tool writes:

- `processed/aq_water_stress/hydro/master_metrics_by_basin.csv`
- `processed/aq_water_stress/hydro/master_metrics_by_sub_basin.csv`

These are the canonical Aqueduct-on-SOI hydro master files consumed by the dashboard.

It also writes QA files under `IRT_DATA_DIR/aqueduct/`, including coverage diagnostics for the generated hydro masters.

## Step 7: Make hydro summary behavior safe in the dashboard

During integration we found that Hydro `Sub-basin` mode with:

- a selected basin
- `Sub-basin = All`

was incorrectly routed into the admin state-summary view, which expects a `state` column and crashed on hydro data.

This was fixed by introducing:

- [`india_resilience_tool/app/views/hydro_summary_view.py`](../india_resilience_tool/app/views/hydro_summary_view.py)

and updating the right-panel routing so basin-wide sub-basin selections produce a hydro-aware basin summary instead of an admin summary.

## Step 8: What the dashboard is actually visualizing

The dashboard now visualizes:

- SOI basin polygons in basin view
- SOI sub-basin polygons in sub-basin view

The displayed values are:

- Aqueduct values cleaned on HydroSHEDS `pfaf_id`
- then area-weighted onto SOI hydro polygons

So the geometry and the data come from different sources:

- geometry source: SOI hydro polygons
- value source: Aqueduct HydroSHEDS-based water-stress products
- transfer bridge: the equal-area overlap crosswalk

This is the intended design.

## Step 9: Interpretation and caveats

### Delta interpretation

The dashboard delta is computed as:

- selected scenario value minus historical baseline

For Aqueduct, this is technically correct as a scenario-versus-baseline change in Aqueduct water stress.

It should **not** be loosely described as a pure climate-only delta. Aqueduct scenario values are broader than a climate-only signal.

### Coverage matters

Some SOI hydro units have excellent Aqueduct overlap coverage, while others have very low coverage.

The QA files should be used when validating edge cases:

- high-coverage units are the best places to audit value behavior
- very low-coverage units should be interpreted more carefully

### Scope limits of the current tranche

Current Aqueduct onboarding is limited to:

- water stress
- hydro units only
- area-weighted transfer from HydroSHEDS L6 to SOI hydro

It does not yet include:

- district/block transfer
- Aqueduct yearly trend files in the dashboard format
- a broader Aqueduct metric family beyond the current water-stress onboarding tranche

## How to Read the Validation Package

The validation outputs under `IRT_DATA_DIR/aqueduct/validation/` are meant to be read in a specific order.

### Recommended reading order

1. `validation_summary.md`
2. `field_semantics_audit.md`
3. `pfaf_cleaning_checks.csv`
4. `coverage_reliability_basin.csv` and `coverage_reliability_subbasin.csv`
5. `crosswalk_conservation_basin.csv` and `crosswalk_conservation_subbasin.csv`
6. `projection_sensitivity_summary.csv`
7. `master_value_spotcheck.csv`
8. `sample_audit_units.csv`

This order moves from high-level interpretation to increasingly detailed technical audit.

### What each file answers

`validation_summary.md`

- start here
- gives the headline counts:
  - cleaned `pfaf_id` status
  - reliability tiers
  - spotcheck reproducibility
  - largest conservation deviations
  - largest sensitivity differences

`field_semantics_audit.md`

- answers: are we comparing the intended Aqueduct fields?
- confirms that:
  - baseline comes from `bws_raw`
  - future comes from `*_ws_x_r`
- use this file whenever someone questions what the dashboard delta actually means

`pfaf_cleaning_checks.csv`

- answers: did the cleaned Aqueduct layer build correctly?
- use it to inspect:
  - one-row-per-`pfaf_id` integrity
  - geometry presence and validity
  - geometry equality between cleaned baseline and clean future geometry

`coverage_reliability_*.csv`

- answers: how trustworthy is each SOI basin or sub-basin result?
- these are target-side QA files
- they tell you how much of the SOI unit is covered by Aqueduct overlap
- use the `reliability_tier` column first:
  - `high`
  - `moderate`
  - `low`

`crosswalk_conservation_*.csv`

- answers: how much of each Aqueduct `pfaf_id` is actually represented inside the SOI hydro system?
- these are source-side QA files
- they are different from coverage:
  - coverage asks whether the target SOI unit is well supported
  - conservation asks whether the source Aqueduct polygon is mostly retained in the transfer

`projection_sensitivity_summary.csv`

- answers: how stable are the results if we vary the transfer method?
- use this file to distinguish:
  - very small differences for `alt_equal_area_vs_epsg6933`, which support the chosen equal-area projection
  - potentially larger differences for `dominant_overlap_vs_weighted`, which show why area-weighted transfer is preferable to winner-takes-all assignment

`master_value_spotcheck.csv`

- answers: can the final basin/sub-basin master values be reproduced exactly from the crosswalk and source values?
- this is the most concrete reproducibility file
- for a technical review, this is the file to use when tracing a final dashboard value back to its contributing `pfaf_id`s

`sample_audit_units.csv`

- answers: which units should be used for manual GIS audit?
- the sample is chosen to include high-, moderate-, and low-reliability cases

### How to interpret the main result types

Use these distinctions consistently:

- semantic audit:
  - are we using and interpreting the right Aqueduct fields?
- cleaning integrity:
  - did the `pfaf_id` cleanup produce a stable HydroSHEDS source layer?
- target reliability:
  - is a basin or sub-basin well covered by Aqueduct overlap?
- source conservation:
  - is a `pfaf_id` mostly retained inside the SOI hydro system?
- sensitivity:
  - do results change materially under alternate reasonable transfer choices?
- spotcheck reproducibility:
  - do the final written master values numerically match direct recomputation?

These are complementary checks. A low-conservation `pfaf_id` near a boundary does not automatically invalidate a high-coverage SOI target unit, and vice versa.

### Practical green / yellow / red reading rule

Green:

- `pfaf_id` cleaning checks are all or nearly all `ok`
- sampled spotcheck differences are effectively zero
- most target units are `high` reliability
- alternate equal-area projection differences are negligible

Yellow:

- some basins or sub-basins fall into `moderate` or `low` reliability
- a few source `pfaf_id`s show partial conservation
- some units are sensitive to dominant-overlap versus weighted transfer

Red:

- many non-OK cleaned `pfaf_id` rows
- large spotcheck mismatches
- many low-reliability target units
- large differences even across reasonable equal-area projection choices

### Practical rule for downstream interpretation

For the current Aqueduct tranche:

- `high` reliability units are the strongest candidates for technical discussion and dashboard use
- `moderate` reliability units should be used with explicit caveats
- `low` reliability units should not be overclaimed and should be checked manually before being used in important interpretation

When in doubt:

- read `field_semantics_audit.md` first for interpretation
- use `master_value_spotcheck.csv` to reproduce a reported value
- use `sample_audit_units.csv` and GIS overlays for manual review

## End-to-end processing sequence

The full Aqueduct onboarding sequence is:

1. Inspect raw Aqueduct layers in GIS.
2. Identify `future_annual` as the clean HydroSHEDS geometry source.
3. Read baseline values from CSV and normalize them to one row per `pfaf_id`.
4. Build `baseline_clean_india.geojson` and its QA CSV.
5. Export `future_annual_india.geojson`.
6. Intersect clean Aqueduct HydroSHEDS polygons with SOI basins and sub-basins in `EPSG:6933`.
7. Build durable Aqueduct → SOI overlap crosswalk CSVs.
8. Use overlap area as the transfer weight to compute SOI basin and sub-basin Aqueduct masters.
9. Register Aqueduct as a hydro-only water-risk metric in the dashboard.
10. Fix summary-view routing so hydro selections behave correctly in the right panel.
